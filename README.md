# 企业内部HTTP通知投递服务

企业内部通用HTTP通知投递服务，屏蔽第三方API差异，为业务系统提供可靠、统一的通知投递能力。

---

## 一、问题理解
### 业务痛点
企业内部多个业务系统在关键事件发生时需要调用外部第三方HTTP API进行通知，存在以下痛点：
1. **重复开发**：每个业务系统都需要独立对接第三方API，重复实现签名、重试、错误处理等逻辑
2. **可靠性差**：业务系统往往只做简单调用，缺乏完善的重试、持久化、监控能力，通知送达率无法保障
3. **变更困难**：第三方API变更时，所有对接的业务系统都需要同步修改，维护成本高
4. **可观测性弱**：缺乏统一的投递状态查询、日志审计、监控告警能力，问题排查困难

### 核心目标
作为通用投递层，**屏蔽第三方差异，提供可靠投递能力**，让业务系统只需要关注业务逻辑，不需要关心通知投递的细节。

---

## 二、整体架构与核心设计
### 核心架构
```
上游业务系统 → HTTP API → 幂等校验 → 持久化存储 → 首次投递尝试
                                      ↓
                           定时重试调度 ← 格式转换 ← 限流校验 ← 投递执行 → 第三方API
                                      ↓
                                 死信队列 → 人工干预
```

### 核心流程
1. **接收流程**：上游调用 → 认证校验 → 幂等校验 → 持久化到数据库 → 异步发起首次投递 → 返回响应
2. **投递流程**：查询目标系统配置 → 模板渲染请求头/请求体 → 发送HTTP请求 → 成功标记完成/失败更新重试信息
3. **重试流程**：定时任务扫描待重试消息 → 批量执行重试 → 超过最大重试次数进入死信队列
4. **死信处理**：死信告警通知管理员 → 支持人工重发/丢弃/导出处理

### 核心特性
- ✅ **统一接口**：标准化HTTP接口供内部业务系统提交通知
- ✅ **格式适配**：自动将内部格式转换为第三方要求的请求格式
- ✅ **可靠投递**：持久化存储 + 指数退避重试 + 死信队列，保证至少送达一次
- ✅ **流量隔离**：不同第三方API限流隔离，避免故障扩散
- ✅ **幂等保证**：通过业务ID实现幂等，避免重复投递
- ✅ **状态查询**：提供投递状态查询接口
- ✅ **配置化**：目标系统配置化管理，无需代码修改即可接入新第三方

## 三、关键工程决策与取舍说明
### 系统边界决策
| 能力 | 是否支持 | 决策说明 |
|------|----------|----------|
| 业务合法性校验 | ❌ 不支持 | 保持职责单一，避免耦合业务逻辑，由上游系统保证 |
| 毫秒级超实时投递 | ❌ 不支持 | 可靠投递需要持久化必然有延迟，超实时场景建议业务系统直接调用 |
| 非HTTP协议适配 | ❌ 第一版不支持 | 聚焦核心场景，避免过度复杂，后续可按需扩展 |
| Exactly Once投递 | ❌ 不支持 | 实现成本极高，业务场景可接受少量重复，要求下游幂等即可 |
| 超过1MB大数据传输 | ❌ 不支持 | 通知场景为小数据量，大数据建议通过对象存储传URL |

### 可靠性决策
- **投递语义**：✅ 选择**至少一次（At Least Once）**，优先保证送达，接受少量重复
- **重试策略**：指数退避 + 最大重试次数，仅重试网络错误、5xx、429响应
- **失败兜底**：超过最大重试次数进入死信队列，保留7天，支持人工重发

### 架构取舍（第一版）
| 技术方案 | 是否采纳 | 决策说明 |
|----------|----------|----------|
| 消息队列（RocketMQ/Kafka） | ❌ 不采纳 | 初期流量低，定时任务+MySQL足够简单可靠 |
| 流式处理框架（Flink/Spark） | ❌ 不采纳 | 运维成本高，初期性能需求完全不需要 |
| 微服务架构 | ❌ 不采纳 | 功能简单，单体架构足够，避免不必要的复杂度 |
| 动态脚本转换 | ❌ 不采纳 | 初期接入第三方少，固定模板足够，避免安全风险 |
| 分布式事务 | ❌ 不采纳 | 通知场景只需要最终一致性，分布式事务成本过高 |

> 完整架构决策文档请参考：[architecture-decision.md](./docs/architecture-decision.md)

## 快速开始

执行`docs/migration.sql`创建数据库和表结构：

复制`config/config.yaml`并修改数据库、Redis等配置。

```bash
go mod tidy
go run main.go
```

## API接口

### 提交通知
```http
POST /api/v1/notifications
Content-Type: application/json

{
  "biz_id": "order_paid_123456",
  "biz_type": "order_paid",
  "target_system": "example_system",
  "content": {
    "order_id": "123456",
    "amount": 99.9,
    "pay_time": "2024-01-01 12:00:00"
  }
}
```

响应：
```json
{
  "code": 200,
  "message": "Notification accepted",
  "data": {
    "biz_id": "order_paid_123456",
    "id": 1
  }
}
```

### 查询投递状态
```http
GET /api/v1/notifications/{biz_id}
```

响应：
```json
{
  "code": 200,
  "message": "Success",
  "data": {
    "biz_id": "order_paid_123456",
    "biz_type": "order_paid",
    "target_system": "example_system",
    "status": 1,
    "retry_count": 0,
    "max_retry_count": 3,
    "next_retry_time": null,
    "last_error": "",
    "created_at": "2024-01-01T12:00:00Z",
    "updated_at": "2024-01-01T12:00:01Z"
  }
}
```

状态说明：
- `0`：待投递
- `1`：投递成功
- `2`：投递失败，等待重试
- `3`：死信（超过最大重试次数）

## 配置说明

### 目标系统配置
在`target_system_config`表中配置第三方系统信息：
| 字段 | 说明 | 示例 |
|------|------|------|
| system_code | 系统唯一编码 | `example_system` |
| system_name | 系统名称 | `示例第三方系统` |
| endpoint | API地址 | `https://api.example.com/webhook` |
| method | 请求方法 | `POST` |
| headers | 请求头模板 | `{"Content-Type": "application/json", "Authorization": "Bearer ${secret}"}` |
| body_template | 请求体模板 | `{"event_type": "${biz_type}", "event_id": "${biz_id}", "data": ${content}}` |
| timeout | 请求超时（毫秒） | `5000` |
| rate_limit | 每秒限流 | `100` |

### 模板变量
模板中支持以下变量：
- `${biz_id}`：业务唯一ID
- `${biz_type}`：业务类型
- `${target_system}`：目标系统编码
- `${content}`：通知内容JSON
- `${created_at}`：创建时间

## 重试机制
- 指数退避策略：第1次重试等待1秒，第2次等待2秒，第3次等待4秒
- 仅重试网络错误、5xx响应、429限流响应
- 默认最大重试3次，可配置
- 超过最大重试次数进入死信队列，保留7天，支持人工重发

## 演进规划
### 第一版（当前）
- 单体架构 + MySQL + 定时任务重试
- 支持TPS<100，第三方接口<20个

### 第二阶段
- 引入RocketMQ + XXL-Job分布式调度
- 支持TPS 100-1000，接口20-100个

### 第三阶段
- 微服务拆分 + Nacos配置中心 + 动态脚本
- 支持TPS>1000，接口>100个

### 第四阶段
- 流式处理 + 全局流量调度
- 支持TPS>10000，接口>500个
