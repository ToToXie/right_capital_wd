# 企业内部HTTP通知投递服务 - 架构设计与决策文档

## 一、项目概述
本服务作为企业内部通用通知投递层，屏蔽第三方HTTP API差异，为上游业务系统提供可靠、统一的通知投递能力，解决多业务系统重复对接第三方、投递可靠性无法保障的痛点。

**核心目标**：保证通知稳定可靠送达，上游业务系统无需关心第三方API差异和投递可靠性问题。

---

## 二、系统边界与能力范围
### 2.1 本系统明确解决的问题
| 能力 | 说明 |
|------|------|
| 统一接入接口 | 提供标准化HTTP API，所有上游业务系统只需要对接这一个接口即可完成所有第三方通知投递 |
| 格式自动适配 | 内置模板引擎，自动将内部统一格式转换为不同第三方要求的Header/Body请求格式，屏蔽第三方差异 |
| 可靠投递保障 | 通过持久化存储、重试机制、死信队列等机制，保证通知至少送达一次 |
| 流量控制与隔离 | 针对不同第三方API进行独立限流，避免单个第三方故障扩散影响整个系统 |
| 可观测能力 | 提供投递状态查询接口、全链路日志审计、监控指标埋点，支持问题排查和运营分析 |
| 配置化管理 | 所有第三方API的地址、认证信息、重试策略、限流阈值等配置统一管理，新增接入无需修改代码 |

### 2.2 本系统明确不解决的问题及决策依据
| 不解决的问题 | 决策原因 |
|--------------|----------|
| 通知内容的业务合法性校验 | 保持职责单一，避免耦合上游业务逻辑，业务合法性由上游业务系统保证，本系统只负责投递 |
| 毫秒级超实时投递 | 可靠投递需要持久化存储，必然引入毫秒级以上延迟；超实时场景由业务系统直接调用第三方API即可，本系统专注于可靠投递场景 |
| 非HTTP协议适配 | 第一版聚焦核心场景，避免系统过度复杂；后续可根据实际需求扩展其他协议（如WebService、TCP等） |
| Exactly Once绝对不重复投递 | 实现成本极高（需要分布式事务、双向确认、去重校验等复杂机制），且业务场景普遍可接受少量重复通知，要求下游系统实现幂等性即可解决重复问题 |
| 超过1MB的大数据量传输 | 通知场景普遍为小数据量事件，大数据量场景建议通过对象存储传递URL，避免占用大量存储和带宽资源 |

## 三、可靠性设计与失败处理
### 3.1 投递语义选择
**明确选择：至少一次（At Least Once）**

**选择依据**：
- 业务场景可接受少量重复通知，实现成本低，完全满足"稳定可靠送达"的核心需求
- 下游系统实现幂等性的成本远低于本系统实现Exactly Once的成本
- 避免为了极低概率的重复问题引入过度复杂的架构设计

### 3.2 失败处理策略
#### 3.2.1 短期失败处理
1. **重试机制**：
   - 采用指数退避策略：第1次重试等待1秒，第2次等待2秒，第3次等待4秒，以此类推
   - 默认最大重试3次，可针对不同第三方系统单独配置
   - 重试范围：仅对网络错误、5xx服务端错误、429限流错误进行重试
   - 不重试范围：4xx客户端错误（如参数错误、鉴权失败、资源不存在等），此类错误重试也无法成功，直接进入死信
2. **熔断降级**：
   - 当第三方API失败率超过50%时，自动降级为低频率重试，避免雪崩效应
   - 可配置单个第三方的最大并发请求数，避免打垮下游系统

#### 3.2.2 超过重试次数处理
- 消息进入死信队列，数据默认保留7天，可配置
- 死信队列支持人工查询、手动重发、批量导出等操作
- 可配置告警规则，当死信数量超过阈值时通知管理员介入处理

#### 3.2.3 第三方长期不可用处理
- 消息持久化存储在数据库中，不会丢失，直到人工处理
- 支持配置降级策略，降低重试频率，避免资源浪费
- 支持人工批量导出死信数据，进行离线处理
- 第三方恢复后，支持批量重发所有积压消息

## 四、架构取舍与演进路径
### 4.1 第一版不采纳的过度设计及原因
| 不采纳的设计 | 决策原因 |
|--------------|----------|
| Flink/Spark等流式处理框架 | 初期流量小（TPS<100），复杂框架运维成本高，MySQL+定时任务的方案足够简单可靠，满足初期需求 |
| 分布式事务 | 通知场景只需要最终一致性，分布式事务复杂度高，投入产出比极低 |
| 异地多活架构 | 主从架构已能满足99.9%的可用性要求，多活成本过高，初期完全不需要 |
| 动态脚本格式转换 | 初期接入的第三方接口数量少（<20个），固定JSON模板足够使用，动态脚本引入安全风险和运维复杂度 |
| RocketMQ/Kafka等消息队列 | 初期流量低，定时任务扫描数据库的方案足够简单可靠，引入MQ增加运维复杂度和学习成本 |
| 微服务架构 | 初期功能简单，单体架构足够支撑，微服务拆分带来的运维成本远大于收益 |

### 4.2 未来演进路径（按流量和复杂度增长）
| 阶段 | 规模阈值 | 演进内容 | 收益 |
|------|----------|----------|------|
| 第一版（当前） | TPS<100，第三方接口<20个 | 单体架构 + MySQL + 本地定时任务重试 | 开发快、运维成本低，满足初期需求 |
| 第二阶段 | TPS 100-1000，第三方接口20-100个 | 1. 引入RocketMQ异步解耦收发流程<br>2. 引入XXL-Job分布式调度替代本地定时任务<br>3. 增加Redis缓存配置和限流能力<br>4. 增加监控告警体系 | 提升系统吞吐量，支持分布式部署，满足中等规模需求 |
| 第三阶段 | TPS>1000，第三方接口>100个 | 1. 微服务拆分（接收服务/投递服务/管理服务）<br>2. 引入Nacos配置中心，实现配置热更新<br>3. 支持Groovy动态脚本实现复杂格式转换<br>4. 增加管理后台，支持可视化配置和死信管理 | 提升扩展性，支持复杂场景和大规模接入 |
| 第四阶段 | TPS>10000，第三方接口>500个 | 1. 引入Flink流式处理，提升投递性能<br>2. 全局流量调度，支持多机房部署<br>3. 智能重试和路由策略<br>4. 多活架构，提升可用性 | 满足超大规模企业级场景需求 |

## 五、核心决策总结
| 决策点 | 选择 | 核心理念 |
|--------|------|----------|
| 投递语义 | 至少一次 | 优先保证送达，接受少量重复 |
| 架构选型 | 单体架构 + MySQL | 简单优先，避免过度设计 |
| 扩展性 | 配置化而非动态化 | 初期够用，逐步演进 |
| 可靠性 | 持久化 + 重试 + 死信 | 数据不丢，故障可人工干预 |
| 边界划分 | 只做投递层，不耦合业务 | 职责单一，避免复杂度膨胀 |
