# AI 使用说明文档

---

## 一、AI 提供的关键帮助
本项目开发过程中，AI工具作为高效辅助手段，在以下方面提供了核心价值：

### 1. 架构设计框架输出
从项目初始阶段就输出了完整的系统设计方案，包括：
- 系统边界划分规则
- 可靠性机制设计（重试、死信、熔断等）
- 分阶段演进路径规划
- 核心数据模型设计
帮助快速搭起整个项目的架构框架，无需从零开始思考所有架构细节，大幅提升了架构设计效率。

### 2. 核心代码快速生成
所有核心业务模块的代码均由AI自动生成，包括：
- 项目初始化、目录结构搭建
- DAO数据访问层、Service业务逻辑层
- API接口Handler、参数校验
- 定时重试任务、模板转换引擎
生成的代码规范、逻辑完整、可直接运行，节省了80%以上的重复编码时间。

### 3. 文档体系自动化输出
AI按照企业级规范快速整理输出了全套文档：
- 数据库建表SQL脚本
- API接口使用说明
- 架构决策文档
- README项目说明文档
避免了手动编写大量标准化文档的重复工作。

### 4. 决策依据补充完善
对于各种设计选择，AI都能给出充分的合理性解释：
- 不同技术方案的优劣势对比
- 设计选择的业务场景适配性说明
- 常见架构陷阱提示
帮助完善决策逻辑，让架构设计的理由更充分、更有说服力。

## 二、未采纳的AI建议及原因
开发过程中部分AI建议基于通用性考虑，不符合本项目MVP版本的极简定位，未被采纳：

| 未采纳的AI建议 | 不采纳原因 |
|----------------|------------|
| 引入Redis做配置缓存和限流 | 第一版MVP流量低（TPS<100），直接读数据库完全足够支撑，避免引入不必要的依赖和运维复杂度 |
| 加入API认证鉴权机制（API Key/JWT） | 服务部署在内网可信环境，第一版不需要额外鉴权，放在后续迭代按需实现 |
| 集成Prometheus监控指标和告警规则 | MVP阶段只需要核心功能，监控可以后续补充，避免增加额外开发工作量 |
| 实现死信管理后台接口（查询/批量重发） | 第一版直接操作数据库即可满足死信管理需求，不需要额外开发管理接口 |
| 引入消息队列（RocketMQ/Kafka）解耦收发流程 | 初期流量低，定时任务+MySQL的方案足够简单可靠，引入MQ会大幅提升运维复杂度 |
| 实现分布式限流能力 | 初期接入第三方数量少，限流需求不迫切，后续规模扩大后再实现 |

## 三、自主做出的关键决策及原因
核心架构决策由人工根据实际业务场景判断做出，AI仅作为参考：

### 1. MVP版本极简架构决策
- **决策内容**：第一版完全不用任何中间件（Redis、MQ等），只依赖MySQL，做最简化的单体架构
- **决策原因**：降低部署和运维成本，快速上线验证需求，初期TPS<100的场景下性能完全足够，避免不必要的复杂度。

### 2. 系统边界严格划分
- **决策内容**：坚决不做业务合法性校验、不支持毫秒级超实时投递、不实现Exactly Once绝对不重复投递，严格遵守投递层的单一职责
- **决策原因**：避免职责膨胀，保持系统简单可靠，把复杂问题推给上游/下游更合适的环节解决（业务校验由上游业务系统做，幂等性由下游第三方系统实现）。

### 3. 投递语义明确选择"至少一次"
- **决策内容**：坚决不追求绝对不重复的投递能力，明确要求下游系统实现幂等性
- **决策原因**：Exactly Once投递实现成本极高（需要分布式事务、双向确认等复杂机制），而业务场景完全可以接受少量重复通知，投入产出比极低。

### 4. 固定模板而非动态脚本的格式转换方案
- **决策内容**：第一版只用静态JSON模板做格式转换，不引入任何动态脚本能力
- **决策原因**：避免动态脚本带来的安全风险和运维复杂度，初期接入的第三方数量少（<20个），固定模板完全可以满足需求。

### 5. 分阶段演进路径设计
- **决策内容**：明确四个阶段的演进阈值和对应的架构升级内容，坚决不做超前设计
- **决策原因**：避免过度设计，每个阶段的架构刚好匹配当时的业务规模，平衡架构复杂度和业务收益，不做无意义的技术超前。

## 四、AI 使用总结
1. **AI定位**：本项目中AI是高效的辅助工具，适合处理标准化、重复性、模板化的工作，大幅提升开发效率。
2. **核心判断**：业务场景适配、架构取舍、边界划分等核心决策必须由人根据实际情况做出，AI的通用建议无法替代对具体业务场景的理解。
3. **使用原则**：AI输出的内容需要经过人工校验和筛选，符合项目定位和阶段目标的内容采纳，不符合的坚决舍弃，避免为了"技术先进性"引入不必要的复杂度。
